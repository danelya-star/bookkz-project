\section*{ВВЕДЕНИЕ}
\addcontentsline{toc}{section}{ВВЕДЕНИЕ}

Современный этап развития информационных технологий характеризуется глобальной цифровизацией всех сфер жизнедеятельности общества. Экономика, сфера услуг, торговля и развлечения массово переходят в онлайн-пространство. В этих условиях онлайн-сервисы развиваются не просто динамично, а опережающими темпами, становясь основным инструментом взаимодействия между потребителями и поставщиками. В частности, сфера бронирования отелей, ресторанов, авиабилетов и мероприятий переживает период глубокой технологической трансформации. Пользователи больше не хотят устанавливать десятки разных приложений для каждой отдельной задачи. Возрастает актуальность разработки мультисервисных платформ (супер-аппов и комплексных веб-решений), которые объединяют разрозненные услуги в рамках единого интуитивно понятного интерфейса.

\textbf{Актуальность выбранной темы} курсового проекта обусловлена сразу несколькими факторами. Во-первых, существует четко выраженный запрос со стороны потребителей на наличие интегрированной платформы для бронирования всего необходимого в пределах родного региона (Республики Казахстан). Существующие мировые лидеры рынка (Booking, Airbnb) зачастую предлагают услуги только в сегменте жилья, а локальные игроки — только билеты (Ticketon) или еду (Chocofood). Во-вторых, малому и среднему бизнесу (региональным отелям, ресторанам, организаторам мероприятий) требуется надежная, дешевая в интеграции и современная платформа для цифровизации своих процессов продаж и ведения учета. В-третьих, создание собственной веб-системы на стыке различных сегментов рынка является сложной инженерной задачей, требующей решения вопросов масштабирования, безопасности платежей и оптимизации архитектуры баз данных. 

\textbf{Цель курсового проекта} — проектирование и программная реализация масштабируемой онлайн-платформы «BookKZ», предназначенной для автоматизации процессов поиска, фильтрации и бронирования различных категорий услуг с поддержкой генерации финансовых отчетностей.

Для успешного достижения поставленной цели в рамках данного исследования необходимо решить следующие \textbf{задачи:}
1. Провести детальный системный анализ предметной области, включающий обзор существующих программных решений конкурентов и выявление их недостатков.
2. Изучить и выбрать оптимальный стек технологий (MERN), обосновав его применимость для систем с высокой нагрузкой.
3. Разработать инфологическую и физическую модели базы данных для эффективного хранения информации о пользователях, бронированиях, услугах и транзакциях.
4. Осуществить проектирование пользовательского интерфейса (UI/UX) клиентской части (Frontend) с учетом требований адаптивной верстки.
5. Написать программный код серверной логики (Backend), разработать защищенный программный интерфейс (REST API) и интегрировать алгоритмы аутентификации.
6. Выполнить интеграционное и функциональное тестирование разработанной системы, проанализировать результаты и разработать отчетные документы (включая генерацию PDF).

\textbf{Объектом исследования} является процесс информационного взаимодействия конечных пользователей (клиентов) и поставщиков услуг в рамках электронной коммерции.

\textbf{Предметом исследования} выступают программные методы, алгоритмы, базы данных и архитектурные паттерны, применяемые для разработки клиент-серверных веб-приложений (Full-Stack разработка).

\textbf{Практическая значимость работы} заключается в том, что разработанный программный продукт представляет собой готовую к внедрению информационную систему. Архитектурные решения, реализованные в рамках проекта, могут быть взяты за основу коммерческими организациями для быстрого создания собственных супер-аппов, а инструменты административной аналитики могут применяться для повышения эффективности бизнес-процессов.

\textbf{Структура курсового проекта.} Работа состоит из введения, трех основных глав (теоретической, проектной и практической), заключения, списка использованной литературы и приложений. Общий объем работы соответствует установленным требованиям.

\section{ТЕОРЕТИЧЕСКАЯ ЧАСТЬ}

\subsection{Общие понятия Web-приложений и их архитектура}
Веб-приложение — это разновидность клиент-серверного программного обеспечения, в котором клиентом выступает браузер пользователя, а сервером — специализированный веб-аппаратный комплекс, который принимает запросы, выполняет вычисления и возвращает ответ по протоколу HTTP/HTTPS. За последнее десятилетие концепция веб-разработки эволюционировала от простых статических HTML-документов до сложных, высокоинтерактивных одностраничных приложений (SPA — Single Page Application). 

Одностраничные веб-приложения кардинально меняют пользовательский опыт. При работе с SPA страница загружается лишь единожды, а при дальнейшей навигации по разделам происходит только асинхронный обмен данными с сервером (обычно в формате JSON). Эта технология позволяет интерфейсу реагировать на действия пользователя молниеносно, создавая ощущение работы с полноценным десктопным приложением. В рамках разработки платформы «BookKZ» именно архитектура SPA является ключевой.

Информационная система онлайн-бронирования должна отвечать строгим критериям качества программного обеспечения: надежности (гарантия обработки брони и избежание овербукинга), безопасности (защита персональных данных пользователей и их платежной информации), удобству использования (юзабилити) и расширяемости (возможность добавления новых категорий услуг без переписывания архитектуры). 

\subsection{Обзор существующих программных решений}
Для глубокого понимания требований к платформе «BookKZ» был проведен сравнительный анализ ведущих мировых и локальных систем бронирования. Рассматривались как монопольные глобальные площадки, так и агрегаторы.

\textbf{Booking.com} — крупнейшая мировая система онлайн-бронирования отелей. Ее главными преимуществами являются огромная база данных жилья по всему миру, многоязычность, продвинутая система отзывов и гибкие условия отмены. Однако недостатком для местного сегмента бизнеса являются высокие комиссии для владельцев объектов размещения и сложность интерфейса из-за перегруженности рекламными баннерами.

\textbf{Airbnb} — платформа, ориентированная на краткосрочную аренду частного жилья. Преимуществом является отличный и современный UX-дизайн, надежная система верификации личности и отзывов. К минусам можно отнести отсутствие возможности бронировать сопутствующие услуги (например, трансфер или столик в ресторане).

\textbf{Chocolife / Ticketon (Казахстан)} — локальные лидеры в сегменте купонов и билетов на мероприятия. Эти системы отлично интегрированы с банковскими приложениями (Kaspi, Halyk), поддерживают локальные реалии и местную валюту (тенге). Однако их функционал сильно фрагментирован. Ticketon позволяет купить билет в кино или театр, но не позволяет забронировать отель. 

Анализ конкурентной среды показал, что ниша интегрированной системы «Travel \& Leisure» (Путешествия и досуг), которая объединяла бы бронирование отелей, ресторанов, мероприятий и транспортных услуг в едином удобном окне для казахстанского рынка, остается недостаточно освоенной. Разрабатываемая платформа «BookKZ» призвана закрыть этот пробел.

\subsection{Выбор архитектуры системы}
Платформа «BookKZ» проектируется на основе микросервисо-подобной монолитной архитектуры с разделением на Frontend и Backend (REST API). REST (Representational State Transfer) — архитектурный стиль взаимодействия компонентов распределённого приложения в сети, который строится на основе базовых HTTP-методов (GET, POST, PUT, DELETE).
Подобное разделение дает следующие преимущества:
\begin{enumerate}
    \item Независимость разработки: команды Frontend и Backend могут работать параллельно, опираясь на задокументированные контракты (API).
    \item Кроссплатформенность: единый Backend может обслуживать как веб-сайт, так и будущие мобильные приложения (iOS, Android).
    \item Масштабируемость: балансировщики нагрузки могут распределять трафик только на нагруженные узлы (например, узел поиска услуг масштабируется отдельно от узла генерации PDF).
\end{enumerate}

\subsection{Анализ программных средств и обоснование выбора стека (MERN)}
Успех современного программного продукта во многом зависит от выбора правильного стека технологий. Стек MERN (MongoDB, Express, React, Node.js) был выбран в качестве фундамента платформы по ряду объективных причин.

\textbf{Node.js} — серверная среда выполнения JavaScript. В её основе лежит событийно-ориентированная (Event-Driven) неблокирующая архитектура ввода-вывода (Non-blocking I/O). Это позволяет Node.js выдерживать тысячи одновременных подключений к серверу без существенных затрат оперативной памяти, что критически важно для платформы бронирования, где пользователи массово занимаются поиском информации.

\textbf{Express.js} — минималистичный и гибкий веб-фреймворк для Node.js, который предоставляет обширный набор функций для разработки веб- и мобильных приложений. Он позволяет легко реализовывать маршрутизацию (Routing), работу с HTTP-заголовками, обработку ошибок и подключение промежуточных слоев (Middleware).

\textbf{React.js} — библиотека JavaScript с открытым исходным кодом для разработки пользовательских интерфейсов. Главной особенностью React является использование виртуального DOM (Virtual Document Object Model), алгоритмы которого рендерят в браузере только те элементы, которые фактически изменились, экономя ресурсы компьютера клиента. Компонентный подход React позволяет разработать элементы (карточка услуги, кнопка бронирования) всего один раз и использовать их по всему приложению.

\textbf{MongoDB} — документо-ориентированная база данных NoSQL. В отличие от классических реляционных СУБД (например, MySQL или PostgreSQL), MongoDB хранит данные не в строгих таблицах, а в гибких BSON-документах (аналогичных JSON). В случае платформы «BookKZ» услуги очень различаются: отель может иметь поля «количество звезд» и «бассейн», в то время как билет на концерт — поля «артист» и «ряд/место». MongoDB позволяет хранить столь разнородные данные в рамках одной коллекции «Services», что существенно упрощает разработку и масштабирование БД.

\section{ПРОЕКТИРОВАНИЕ ПРОГРАММНОГО ПРОДУКТА}

\subsection{Проектирование функциональных требований к системе}
На этапе проектирования были четко определены функциональные бизнес-требования к программному продукту. Приложение должно обслуживать три основные группы пользователей: неавторизованные пользователи (гости), авторизованные клиенты и администраторы.

Для \textbf{гостей} предусмотрен следующий функционал:
- Свободный просмотр каталога всех услуг с использованием строки поиска.
- Фильтрация и сортировка результатов (по категории: отель, ресторан, мероприятие, транспорт; по цене, по дате заезда и выезда).
- Просмотр подробной информации о конкретной услуге (описание, фотографии, рейтинг, отзывы других клиентов).
- Прохождение процесса регистрации аккаунта и авторизации в системе.

Для \textbf{авторизованных клиентов}:
- Все возможности гостя.
- Бронирование выбранной услуги на определенные даты с указанием числа гостей.
- Имитация защищенного процесса онлайн-оплаты заказа.
- Доступ к личному кабинету («Мои бронирования»).
- Скачивание электронного чека в формате PDF для оплаченных бронирований.
- Право оставления текстового отзыва и выставления оценки (от 1 до 5 звезд) после завершения услуги.

Для \textbf{администраторов платформы}:
- Полный доступ (CRUD) к управлению каталогом услуг (добавление, редактирование, сокрытие, удаление услуг из базы).
- Управление базой пользователей (просмотр контактных данных, изменение ролей).
- Просмотр всех транзакций и бронирований системы, возможность отмены мошеннических операций.
- Доступ к дашборду – визуальной аналитике платформы, содержащей графики доходов по месяцам, процентное соотношение продаж по категориям и статистику регистраций.

\subsection{Разработка архитектуры базы данных (MongoDB)}
Разработка структуры базы данных является одним из самых важных этапов, так как от нее зависит производительность приложения. Было спроектировано четыре основные коллекции (Collections):

\textbf{1. Коллекция Users (Пользователи)} 
Содержит поля:
- \textit{name, email, password} (хэшированный пароль).
- \textit{phone, role} (имеет значения "user" или "admin").
- \textit{createdAt} (временная метка создания аккаунта).

\textbf{2. Коллекция Services (Услуги)}
Содержит информацию о доступном инвентаре:
- \textit{title, description, category} (hotel, restaurant, event, transport).
- \textit{price} (числовое значение цены за ночь или билет).
- \textit{location, image, amenities} (массив преимуществ).
- \textit{rating, reviewCount} (вычисляемые поля, кэширующие рейтинг для быстрого поиска).
- \textit{capacity} (вместимость или количество доступных билетов).
- \textit{isActive} (логический флаг видимости услуги для клиентов).

\textbf{3. Коллекция Bookings (Бронирования)}
Связующая таблица транзакций. Каждая запись содержит:
- Ссылку на \textit{userId} и \textit{serviceId} (Foreign Keys в парадигме NoSQL реализованы через логические ссылки ObjectID).
- \textit{checkIn, checkOut} (даты брони).
- \textit{guests, totalPrice}.
- \textit{status} ("pending", "confirmed", "cancelled", "completed").
- \textit{paymentStatus} ("unpaid", "paid", "refunded").
- \textit{paymentTransactionId} (строка транзакции банка).

\textbf{4. Коллекция Reviews (Отзывы)}
Хранит мнения пользователей:
- \textit{user} (автор), \textit{service} (к какой услуге прикреплен).
- \textit{rating} (от 1 до 5).
- \textit{comment, createdAt}.

База данных нормализована до достаточной степени, чтобы не было избыточности, но при этом данные, необходимые для мгновенной отрисовки карточек (рейтинг услуги), хранятся прямо в документе Service.

\subsection{Проектирование пользовательского интерфейса (UX/UI)}
Дизайн-система платформы была тщательно продумана. Интерфейс выполнен в современном минималистичном стиле «Glassmorphism» с использованием темно-синей темы (Dark Mode), которая меньше утомляет зрение пользователей. 

Цветовая палитра:

\subsection{Проектирование механизмов безопасности}
Учитывая, что система работает с коммерческими данными, была реализована многоуровневая модель безопасности:
- \textbf{Аутентификация API}: Каждый запрос к защищенному маршруту сопровождается валидацией JWT токена, генерируемого сервером.
- \textbf{Защита от межсайтового скриптинга (XSS) и подделки запросов (CSRF)}: Токены передаются исключительно через защищенные куки (httpOnly cookie), доступ к которым закрыт для клиентского JavaScript.
- \textbf{Хэширование паролей}: Ни один пароль не хранится в открытом виде. Использован алгоритм `bcryptGenSalt(10)`, делающий метод подбора паролей (Brute-force) вычислительно нерентабельным для злоумышленника.

\subsection{Реализация серверной логики (Node.js Backend)}
Разработка Backend-а была выполнена в папке `server` с использованием Express. Сервер инициализируется на порту, указанном в файле конфигурации `.env`. Интеграция с базой MongoDB осуществляется с помощью Object Data Modeling (ODM) библиотеки \textbf{Mongoose}.

Особого внимания заслуживает сложная архитектура получения статистики для административной панели. Вместо того чтобы сервер запрашивал миллионы записей из БД и обрабатывал их в оперативной памяти (что приведет к зависанию), был применен метод \textbf{MongoDB Aggregation Framework}. 
Ниже представлен фрагмент кода (`adminController.js`), который демонстрирует, как сервер молниеносно агрегирует доходы по месяцам:

\begin{verbatim}
const monthlyRevenue = await Booking.aggregate([
    { $match: { paymentStatus: 'paid' } },
    {
        $group: {
            _id: { $month: "$createdAt" },
            total: { $sum: "$totalPrice" }
        }
    },
    { $sort: { "_id": 1 } }
]);
\end{verbatim}

Этот пайплайн (Pipeline) фильтрует только успешные транзакции, затем группирует суммы покупок по месяцам календарного года и сортирует их по хронологии. В результате сервер возвращает клиенту крошечный JSON-объект идеального формата для построения графика, минимизируя нагрузку на сеть.

\subsection{Разработка клиентского приложения (React.js Frontend)}
Фронтенд-приложение инициализировано с помощью сборщика Vite, что обеспечило высокую скорость горячей перезагрузки (Hot Module Replacement) модулей в процессе разработки. Маршрутизация (Routing) внутри SPA построена поверх библиотеки \textit{react-router-dom}.

Разработаны следующие основные компоненты и страницы:
\textit{Navbar.jsx} (Навигация), \textit{HomePage.jsx} (Лендинг), \textit{ServicesPage.jsx} (Сложный каталог с фильтрами), \textit{ServiceDetailPage.jsx} (Детальная карточка с отзывами и выбором дат заезда). 

Особое внимание уделено UX: при переходе по прямым ссылкам из подвала страницы (например «Мероприятия»), приложение отслеживает URL-параметры `searchParams` через хук `useEffect`, моментально обновляя стейт (State) компонента и делая новый fetch-запрос в API без перезагрузки всей вкладки.

Управление контекстом (состоянием авторизации пользователя) реализовано с помощью React Context API (`AuthContext.js`), что позволяет компоненту «Корзина» или «Оставить отзыв» знать, зашел ли человек в свой аккаунт, и блокировать функционал, выводя всплывающее уведомление библиотеки Toast.

В данном приложении приведен фрагмент исходного программного кода на языке JavaScript, демонстрирующий логику перехвата параметров адресной строки (searchParams) в компоненте \texttt{ServicesPage.jsx} и реализацию эффектов жизненного цикла React. Данный код защищает систему от рассинхронизации пользовательского интерфейса при навигации через подвал сайта.

\vspace{0.5cm}

\begin{verbatim}
import { useState, useEffect, useCallback } from 'react';
import { Link, useSearchParams } from 'react-router-dom';
import { servicesApi } from '../services/api';

export default function ServicesPage() {
    const [searchParams, setSearchParams] = useSearchParams();
    const [services, setServices] = useState([]);
    const [loading, setLoading] = useState(true);
    const [category, setCategory] = useState(
        searchParams.get('category') || ''
    );

    const fetchServices = useCallback(async () => {
        setLoading(true);
        try {
            const res = await servicesApi.getAll({ category });
            setServices(res.data.services);
        } catch (err) {
            console.error(err);
        } finally {
            setLoading(false);
        }
    }, [category]);

    // Хук для отслеживания изменений URL параметров
    useEffect(() => {
        const urlCategory = searchParams.get('category');
        if (urlCategory !== null && urlCategory !== category) {
            setCategory(urlCategory);
        }
    }, [searchParams]);

    useEffect(() => { fetchServices(); }, [fetchServices]);

    return (
        <div className="services-grid">
            {loading ? <p>Loading...</p> : services.map(s => (
                <div key={s._id} className="service-card">
                    <h3>{s.title}</h3>
                </div>
            ))}
        </div>
    );
}
\end{verbatim}

exports.getAdminStats = async (req, res) => {
    try {
        const totalUsers = await User.countDocuments();
        const totalServices = await Service.countDocuments();

        // Агрегация доходов по месяцам (Mongoose)
        const monthlyRevenue = await Booking.aggregate([
            { $match: { paymentStatus: 'paid' } },
            {
                $group: {
                    _id: { $month: "$createdAt" },
                    total: { $sum: "$totalPrice" }
                }
            },
            { $sort: { "_id": 1 } }
        ]);

        // Форматирование данных для фронтенд-библиотеки Recharts
        const formattedRevenue = monthlyRevenue.map(item => ({
            name: new Date(2026, item._id - 1, 1)
                     .toLocaleString('ru-RU', { month: 'short' }),
            total: item.total
        }));

        res.json({
            stats: {
                totalUsers,
                totalServices,
                monthlyRevenue: formattedRevenue
            }
        });
    } catch (error) {
        res.status(500).json({ message: "Server Error" });
    }
};
\end{verbatim}
